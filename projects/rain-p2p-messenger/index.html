Overview

What I built

Private peer-to-peer chat app for Android and Windows. Rain focuses on accepted friends, direct chat, visible connection state, one-to-one file transfer, voice calls, and video calls. The system separates Flutter UI, Riverpod state, runtime controllers, local Drift persistence, realtime signaling, and WebRTC data/media transport.

Built a focused private-communication product surface with explicit friend, route, call, file, and recovery states.

Current maintained targets are Android phones and Windows desktop. Working feature areas include username sign-in, friend search/requests/blocking, peer chat over WebRTC data channels, connection diagnostics, one-to-one file transfer, and one-to-one voice/video calls. Push call wakeups, group calls, and app-store packaging are not claimed.

Highlights

What makes it notable

Explicit connection states

Friend requests, accepted peers, blocked users, call states, chat states, and transfer progress are visible instead of hidden behind vague status text.

Focused peer workflows

Direct chat, one-to-one file transfer, voice calls, and video calls are presented as maintained feature areas.

Separated runtime layers

UI, local persistence, signaling, session policy, and WebRTC transport are split into focused packages and app layers.

Engineering

Decisions and constraints

Decision 01

Transport and UI separation

WebRTC data and media behavior stays separate from Flutter screen code so chat, calls, and file transfer can evolve independently.

Decision 02

Local persistence for communication state

Identity, friends, messages, and files are tracked locally with Drift-backed state instead of making the UI depend on transient runtime state only.

Decision 03

Clear product claims

The case study avoids claiming push call wakeups, group calls, or app-store packaging that are not part of the maintained scope.
